# 文字起こし組み込み＋チケット制モデル

## 背景

現在のjimacraftは「ユーザーがGoogle AI Studioで文字起こし → テキストをアプリに貼り付け → SRT生成」という流れ。
プロンプトを公開しているため、ノウハウが丸見えで、技術者なら自力で完結できてしまう。

## 新しい構想

### 音声文字起こしをアプリ内に組み込む

- WhisperX（Whisper + pyannote）をクラウドGPUで実行
- ユーザーは音声ファイルをアップロードするだけ
- タイムスタンプ＋話者分離＋テキストが一発で出力される
- プロンプトやモデル設定はサーバー側に隠蔽 → ノウハウが外に出ない

### 課金モデル: 月額制 → チケット制に移行

**有料（チケット消費）**: 音声 → 文字起こし（GPU処理）
**無料（無制限）**: テキスト → SRT変換 → マーカー編集 → 話者別分割 → ダウンロード

### なぜこの構造が機能するか

- 無料部分を使うには「文字起こし済みテキスト」が必要
- それを自力で用意するにはWhisperXセットアップ、GPU環境構築、出力整形が必要 → 非常に面倒
- 「入力データの調達コスト」が暗黙の壁となり、無料機能を制限なしで提供しても価値が守られる
- 機能ロックではなく「便利さ」で課金を促すので、ユーザーの心理的抵抗が小さい

## チケット制の設計案

### 基本ルール
- 1チケット = 1回の文字起こし処理
- 同じ音声ファイルは3回までリトライ無料（精度に不満があるときのやり直し用）
- 3回分のリトライコストを吸収できる価格設定にする

## 月額制と比較したメリット

| 観点 | 月額制 | チケット制 |
|------|--------|-----------|
| 解約リスク | 使わない月に解約される | 買った分は残るので解約がない |
| コスト構造 | 固定収入だがGPU費用は変動 | 使用量に連動、赤字リスク低 |
| まとめ買い | なし | 先にまとまった売上が入る |
| ユーザー心理 | 「今月使ってないのに払ってる」 | 「必要なとき使える」 |
| プラン複雑さ | Free/Plus/Proの3段階 | チケット購入のみ、シンプル |

## プライバシー

- 音声ファイルはサーバーレスGPU（Modal, RunPod等）で処理
- 処理完了後、音声ファイルは即時削除
- 結果（テキスト）のみ返却
- サーバーに音声は保存されない設計

## 技術構成（検討中）

- 文字起こしサーバー: Modal or RunPod（サーバーレスGPU）
- モデル: WhisperX（Whisper large-v3 + pyannote）
- 連携: jimacraft Web UI → API → GPUサーバー → 結果返却
- 音声形式: MP3, MP4, WAV等に対応
